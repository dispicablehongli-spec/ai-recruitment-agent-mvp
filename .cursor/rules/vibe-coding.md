# Vibe Coding Rule

## 读文档策略
- 优先读当前 milestone 文档 + 相关契约文档，不整篇回读主 plan。
- 在需求冲突、边界不清、或你要求全局复核时再回读主 plan。

## 开工前
- 声明本次使用的文档清单（2-4 个文件），核对实现与目标、边界、验收标准一致。
- 若当前 milestone 缺少目标、验收或边界任一项，先发起最多 5 个澄清问题。

## 卡住处理
- 同一问题连续 2 次失败时停下，复盘已尝试方案，提出新拆解或 MVP fallback。

## 里程碑收尾
- 验收通过后提示你提交，并建议打里程碑标签；提交需你确认。
- 需求变更先更新文档，再改代码。
- 凡「值得记录的交付或修复」，按下方 **版本日志** 更新 `docs/plan-version-log.md`（与用户确认是否发版、是否打 tag）。
- **用户要求提交 Git / push / 同步到 GitHub 时**：除非用户明确说 **不必改日志**，否则须在 **同一轮** 内先按下方 **版本日志** 写好本次改动的摘要（新日期节或按规则追加），把 `docs/plan-version-log.md` 纳入提交，再 `commit` / `push`。

## 版本日志（`docs/plan-version-log.md`）

体例对齐 [Keep a Changelog](https://keepachangelog.com/)（**Added** / **Changed** / **Fixed**）。更新时 **打开该文件**，在其 **版本索引** 与正文末尾 **当前能力快照** 处按下列规则操作。

### 必须遵守

- **版本日期**：以用户消息或会话资料中的 **Today's date** 为准；用户指出日期错误时合并/更正版本节，避免未确认时使用「次日」。
- **追加，不篡改**：已发布日期的小节原则上不改（错别字除外）；新内容用新日期新增一节 `## YYYY-MM-DD — 简短标题`。
- **版本索引表**：新增一行（日期、可选 Git 标签、一句话说明）。
- **小节结构**：该日期下使用 `### Added` / `### Changed` / `### Fixed`（可省略空类）；Post-Plan 条目可保留 **P1.x / P2.x** 等编号以便与主 Plan §15 对照。
- **主 Plan §15**：若在 Cursor 主 Plan 中维护了 Post-Plan（§15），将等价要点同步到 **同一天或对应发布节**，避免仓库与主 Plan 长期不一致。
- **当前能力快照**：文档末尾 **截至 YYYY-MM-DD** 的 In scope / Out of scope 在**每次新增发布节时**更新为「截至最新版本」。

### 可选

- 发版时在 Git 打语义化标签（如 `v0.1.0`），并在该节的 **标签** 与索引表「标签」列写明。
- 若变更仅镜像主 Plan 某一节，在条目中标注章节（如 §15）。

### 新版本节模板（复制到新日期节）

```markdown
## YYYY-MM-DD — 简短标题

**标签**：（可选）`v0.x.y`
**关联主 Plan**：（可选）章节或里程碑 ID

### Added
- …

### Changed
- …

### Fixed
- …
```
